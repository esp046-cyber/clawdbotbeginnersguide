üéÆ Understanding Claude's Capabilities & Limitations

Set realistic expectations for what Claude can and cannot do

‚úÖ What Claude CAN Do (Really Well!)

1. Create Complete Web Apps

  • Fully functional apps that work in your browser
  • Beautiful designs with colors, layouts, and animations
  • Interactive features like buttons, forms, and calculations
  • Mobile-responsive designs that work on any device

2. Understand Plain English

  • You don't need to know coding terminology
  • Describe things in your own words
  • Claude figures out what you mean
  • Example: "Make it pretty" = Claude adds nice styling

3. Make Changes Instantly

  • Modify colors, sizes, layouts on demand
  • Add new features to existing apps
  • Fix bugs and errors
  • Optimize for different screen sizes

4. Build Different Types of Apps

  • Calculators: BMI, mortgage, tip, currency conversion
  • Trackers: Expenses, habits, workouts, time
  • Games: Quizzes, memory games, word games
  • Tools: Password generators, timers, color pickers
  • Organizers: Todo lists, calendars, planners
  • Visualizations: Charts, graphs, dashboards

5. Explain How Things Work

  • Ask "How does this feature work?" and Claude explains
  • Learn as you build
  • Get simplified explanations without jargon

‚ùå What Claude CANNOT Do

1. Remember Previous Conversations

Each chat is independent. If you start a new chat, Claude won't remember the app from your last conversation. Solution: Keep working in the same chat, or save your apps!

2. Access the Internet or External Data

Claude can't pull live data from websites (like real-time stock prices or weather). However: Claude can create apps that YOU can connect to data sources, or use built-in APIs.

3. Create Native Mobile Apps

Claude creates web apps (HTML) that work in browsers, not iPhone/Android apps you download from app stores. But: These web apps work great on mobile browsers!

4. Create Apps That Need Servers

Claude creates "client-side" apps that run in your browser. It can't create apps that need databases or server backends. However: It can use browser storage for saving data locally!

5. Make Apps "Go Viral" or Handle Millions of Users

Claude creates personal apps, not enterprise-scale applications. Perfect for personal use, small teams, prototypes, and learning!

üîß Debugging Guide: How to Fix When Things Go Wrong

Step 1: Identify the Problem

Ask Yourself These Questions:

  1. What were you trying to do? "I clicked the save button"
  2. What did you expect to happen? "The note should save"
  3. What actually happened? "Nothing happened at all"
  4. Can you reproduce it? "Yes, happens every time"

Step 2: Common Problems & Solutions

Problem: Button Doesn't Do Anything

Tell Claude:

"When I click the [BUTTON NAME] button, nothing happens. I expected it to [WHAT IT SHOULD DO]. Can you check if the button is connected properly?"

Problem: App Looks Broken on Mobile

Tell Claude:

"The app looks fine on computer but broken on my phone. The [SPECIFIC ISSUE like buttons overlap, text is tiny, things run off screen]. Can you make it mobile-responsive?"

Problem: Data Doesn't Save

Tell Claude:

"When I close the browser and come back, all my data is gone. Can you add local storage so my [ITEMS] are saved permanently?"

Problem: Colors Are Hard to Read

Tell Claude:

"The text is hard to read because the contrast isn't good. Can you change the [TEXT COLOR] so it stands out better against the [BACKGROUND COLOR]?"

Problem: Feature Works Incorrectly

Tell Claude:

"The calculator gives wrong results. When I enter 5 + 3, it shows 35 instead of 8. Can you fix the addition function?"

Problem: Layout Is Messy

Tell Claude:

"The layout looks cluttered with everything squished together. Can you add more spacing between elements and make it cleaner?"

Step 3: Debugging Workflow

1. Test Methodically

  • Test one feature at a time
  • Try different inputs (numbers, text, special characters)
  • Test on different devices if possible
  • Try edge cases (empty input, very long text, zero, negative numbers)

2. Describe Precisely

The [SPECIFIC ELEMENT] doesn't [SPECIFIC ACTION] when I [WHAT YOU DO].
Steps to reproduce:
1. [STEP 1]
2. [STEP 2]
3. [WHAT HAPPENS]

Expected: [WHAT SHOULD HAPPEN]
Actual: [WHAT ACTUALLY HAPPENS]

3. Let Claude Fix It

Claude will usually:

  • Acknowledge the bug
  • Explain what was wrong
  • Show you the fixed version
  • You can test again!

4. If Still Not Working

Try saying:

"That didn't fix it. Let me explain again with more detail: [VERY DETAILED DESCRIPTION]. Can you try a different approach?"

Prevention Tips

‚úÖ Test after every change - Don't add 5 features then test. Test after each one!
‚úÖ Save working versions - Download the app before making big changes!
‚úÖ Be specific in prompts - Clear instructions = fewer bugs!
‚úÖ Test edge cases - What happens if someone enters zero? Empty text? 100 items?

üéØ Understanding Artifacts: Deep Dive

Everything you need to know about how artifacts work and how to use them effectively

What Exactly Is an Artifact?

Definition

An artifact is a self-contained, interactive preview of code that Claude creates. Think of it as a mini-website or app that appears right in your conversation, letting you see and interact with what Claude built without leaving the chat.

Analogy: If Claude is a chef, the artifact is the dish served on a plate. You can see it, taste it (interact with it), and decide if it needs more seasoning (modifications) - all without going into the kitchen (the code).

Types of Artifacts Claude Can Create

1. HTML Artifacts (.html)

What: Complete web apps with HTML, CSS, and JavaScript all in one file

Best for: Simple to medium complexity apps, calculators, forms, dashboards

Pros: Works everywhere, easy to download and share, no dependencies

Cons: Can get messy with very complex apps

Example: Todo list, budget tracker, timer, calculator

2. React Artifacts (.jsx)

What: Interactive components using React framework with modern features

Best for: Complex interactive apps, data visualization, real-time updates

Pros: More powerful, better for complex state management, modern

Cons: Slightly more technical (but Claude handles it!)

Example: Data dashboards, games with complex state, interactive charts

3. Markdown Artifacts (.md)

What: Formatted text documents with headings, lists, links, images

Best for: Documentation, guides, articles, reports, notes

Pros: Clean, readable, easy to convert to other formats

Cons: Not interactive, just for reading

Example: This guide you're reading, user manuals, blog posts

4. SVG Artifacts (.svg)

What: Scalable vector graphics - images made of mathematical shapes

Best for: Logos, icons, diagrams, illustrations, infographics

Pros: Scales to any size without losing quality, editable

Cons: Better for simple graphics than photos

Example: Company logo, flowchart, organizational chart, custom icons

5. Mermaid Diagrams (.mermaid)

What: Code that generates flowcharts, sequence diagrams, timelines

Best for: Process flows, org charts, system architectures, timelines

Pros: Easy to create complex diagrams with text

Cons: Limited styling options

Example: Project timeline, decision tree, workflow diagram

How Artifacts Work Behind the Scenes

Important Concept: Artifacts Are Sandboxed

Artifacts run in an isolated environment (sandbox) which means:

  • ‚úÖ They can't access your files or personal data (security!)
  • ‚úÖ They can't make network requests to random websites
  • ‚úÖ They work the same for everyone
  • ‚ùå They can't automatically fetch live data from the internet
  • ‚ùå They can't save to your computer without you downloading

Storage & Persistence

Local Storage: When Claude adds "local storage" to your app, data is saved in your browser. This means:

  • Data persists when you close/reopen the artifact in the SAME browser
  • Data is specific to YOUR browser (others can't see it)
  • Data stays even if you download the HTML file and open it elsewhere
  • Clearing browser data will erase it

New Feature: window.storage (2026)

Claude now supports window.storage which is even more powerful:

  • Data persists across sessions in Claude.ai
  • Can be shared between users if specified
  • Survives browser refreshes
  • Perfect for collaborative apps or shared data
Ask Claude: "Use window.storage to save data that persists across sessions and can be shared with others"

üì± Mobile App Best Practices

Essential guidelines for creating apps that work beautifully on phones and tablets

Why Mobile-First Matters

Statistics: Over 60% of web traffic is from mobile devices. If your app doesn't work well on phones, most people won't use it!

Mobile-First Approach: Design for small screens first, then enhance for larger screens. This ensures your app works everywhere!

1. Touch Target Sizes

The 44px Rule

Buttons and interactive elements should be at least 44√ó44 pixels - the size of an average fingertip.

‚ùå Too Small:
Tiny 20px buttons that are impossible to tap accurately
‚úÖ Just Right:
Large 50px+ buttons with comfortable spacing
Ask Claude:
"Make all buttons at least 50px tall and wide enough for easy tapping on mobile phones"

Spacing Between Elements

Add generous spacing between tappable elements to prevent accidental taps.

Minimum 8px spacing between buttons Better: 16-24px spacing for comfort

2. Readable Text

Font Size Guidelines

  • Body text: Minimum 16px (18-20px is better)
  • Buttons: 16-18px
  • Headings: 24px+ for h2, 20px+ for h3
  • Small text/captions: Never below 14px
Ask Claude:
"Use 18px for body text, 24px for headings, and make sure all text is easy to read on mobile"

Line Height & Spacing

Cramped text is hard to read on small screens. Use comfortable line spacing.

Line height: 1.5 to 1.8 for body text Letter spacing: Slightly increase for readability

3. Color Contrast

The Outdoor Screen Test

Mobile phones are often used outside in bright sunlight. Low contrast text becomes invisible!

High Contrast Combinations

‚úÖ Good Contrast:
  • Black text on white background
  • White text on dark blue/navy background
  • Dark gray (#333) on light backgrounds
  • Yellow/white text on dark backgrounds
‚ùå Poor Contrast (Avoid):
  • Light gray text on white background
  • Yellow text on white background
  • Any pastel on pastel combinations
  • Similar shades of the same color
Ask Claude:
"Use high contrast colors - dark text on light backgrounds or light text on dark backgrounds for easy reading in any lighting"

4. Layout & Screen Real Estate

Vertical Scrolling Is Natural

Mobile users are comfortable scrolling vertically. Don't try to cram everything "above the fold."

‚úÖ Single Column Layout: Stack elements vertically for easy scrolling
‚ùå Horizontal Scrolling: Avoid forcing users to scroll sideways - it's awkward on mobile

Thumb-Friendly Navigation

Most people hold phones with one hand and use their thumb. Place important actions within easy reach.

üì± Bottom navigation = Easy thumb access
üì± Top navigation = Requires hand repositioning
Ask Claude:
"Put the main action buttons at the bottom of the screen where they're easy to reach with a thumb"

Avoid Tiny Dropdowns

Traditional dropdown menus are difficult on mobile. Use better alternatives:

  • Radio buttons for 2-5 options
  • Segmented controls for 2-3 options
  • Large tap areas if dropdown is necessary

5. Input & Forms

Mobile-Friendly Input Fields

  • Large input boxes: Minimum 44px height
  • Clear labels: Above or inside input, never too small
  • Appropriate keyboards: Numeric for numbers, email for email
  • Auto-capitalization: Use appropriately (names yes, passwords no)
Ask Claude:
"Make input fields large and easy to tap. Use the numeric keyboard for number inputs and email keyboard for email addresses."

Minimize Typing

Typing on mobile is slower than on desktop. Reduce it wherever possible:

  • Use preset buttons instead of text input when possible
  • Provide suggestions or autocomplete
  • Remember previous inputs
  • Use sliders for numeric ranges

6. Performance & Loading

Fast Loading

Mobile connections can be slower. Keep apps lightweight:

  • Avoid large images (compress them or use SVG)
  • Don't load unnecessary external resources
  • Show loading indicators for any delays > 1 second
  • Progressive loading: Show content as it becomes available

Smooth Animations

Animations should enhance, not slow down the experience:

‚úÖ Quick transitions (200-300ms) ‚úÖ Simple animations (fade, slide) ‚ùå Heavy 3D effects ‚ùå Slow animations (>500ms)

7. Gestures & Interactions

Common Mobile Gestures

  • Tap: Primary action (like clicking)
  • Swipe: Navigate between screens or delete items
  • Pinch: Zoom in/out (if applicable)
  • Pull-to-refresh: Update content (advanced)
Ask Claude:
"Add swipe-to-delete functionality on list items for a mobile-friendly experience"

Don't Rely on Hover

Mobile devices don't have "hover" states. Never hide critical functionality behind hover-only interactions!

‚ùå Delete button only appears on hover
‚úÖ Delete button always visible or appears on tap/swipe

8. Orientation Handling

Portrait vs. Landscape

Most people use phones in portrait, but should work in landscape too:

Ask Claude:
"Make sure the app works in both portrait and landscape orientations on mobile"

9. Testing on Real Devices

Desktop Preview ‚â† Real Mobile Experience

Always test on an actual phone! Things that look fine on desktop may have issues on real devices.

How to Test on Your Phone

  1. Download the HTML file from Claude
  2. Send it to your phone (email, cloud storage, AirDrop)
  3. Open in mobile browser (Safari, Chrome)
  4. Test ALL features with your actual fingers!
  5. Try in different lighting conditions
  6. Test with one-handed use
